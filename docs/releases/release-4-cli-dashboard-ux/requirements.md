# release-4-cli-dashboard-ux — 要件定義

- リリース識別子: `release-4-cli-dashboard-ux`
- ステータス: 確定
- 作成日: 2026-07-02
- 対応する設計: `docs/design/class-diagram.md`（cli-ink レイヤー）

## 背景と目的

release-1〜3 で機能面は完成したが、CLI ダッシュボードの見た目が素朴なテーブル表示に留まっており、Claude Code の Agent View のような TUI らしさに欠ける。また `monomi` 起動直後は静的な1回取得のみで、watch モードを毎回 `w` で手動 ON にする必要があり不便。本リリースはこの2点の UX を改善する。

## スコープの確定（壁打ちでの決定事項）

3つの TUI 方向性（k9s 風ボックス分割・lazygit 風単一大枠・カード型/Agent View 風）をモックアップで提示し、**カード型（Agent View 風）**を選択。続けて「ターミナルの横幅に合わせて広げてほしい」との指示により、**固定1列ではなくレスポンシブな複数列グリッド**を要件に組み込む。

| 論点           | 決定                                                                                                                                                                                       |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 見た目の方向性 | **カード型**（instance 1件 = ボーダー付きボックス、project 名をタイトルに、device/branch/status+age を表示）                                                                               |
| 折返し         | **ターミナル幅に応じて複数列に自動折返し**（狭い端末では1列に縮退）                                                                                                                        |
| watch 既定     | **`monomi` 起動直後から watch モード常時 ON**（従来は初回1回取得のみ、`w` で手動 ON が必要だった）。実装後の実機確認で手動トグルの必要性が無いと判明し、`w` トグル自体を撤去（FR-03 参照） |
| フッター       | 詳細ビュー表示中もフッターが一覧用のままだったため、ビューごとに専用のヒントへ切り替える（FR-04、実機確認で判明し追加）                                                                    |
| 変更しない範囲 | ヘッダー・フィルタバー・フッター・詳細ビュー（Agent View Lv.1）の見た目は今回対象外（現状維持）                                                                                            |

## 機能要件

### FR-01: カード型の instance 一覧表示（優先度: 必須）

- AC-1: 1 instance を1枚のボーダー付きボックスとして描画する。タイトルは project 名、内容は device 名・branch・status（色付きグリフ＋ラベル）+ age を含む
- AC-2: 選択中のカードはボーダー色（例: cyan）で視覚的に区別する（従来の `>` プレフィックス＋反転表示は廃止）
- AC-3: 既存の状態色・グリフ・ラベル・age 整形（`status-display.ts`）をそのまま再利用し、表現ロジックを重複させない
- AC-4: instance が0件のときは「該当するインスタンスがありません」を表示する（現状維持）
- AC-5: `j`/`k`・`↑`/`↓` によるカーソル移動、`Enter` による詳細表示、`1`-`5` フィルタ、`?` ヘルプ、`q` 終了は一覧表示中は全て機能する（`w` watch トグルは FR-03 の改訂により撤去。footer 文言は FR-04 参照）

### FR-02: ターミナル幅に応じたレスポンシブなグリッド折返し（優先度: 必須）

- AC-1: カードは横方向に並び、1行に収まらなくなったら次の行へ自動折返しする（Ink の `flexWrap` を使用）
- AC-2: ターミナル幅が広いほど1行あたりのカード数が増える（例: 40列幅では1列、120列幅では3列程度になることをテストで確認する。厳密な閾値は実装時に決めてよいが、幅に応じて列数が変化することを担保する）
- AC-3: カーソル移動（`j`/`k`/`↑`/`↓`）は既存の1次元インデックス順（`store.filtered()` の並び順）のまま。グリッド上は見た目の折返し順（左から右、上から下）と一致する
- AC-4: 端末幅が取得できない環境（非TTY、テストの `render-to-string` 等）では1列固定にフォールバックする。~~実装当初は列数1のときに `cardWidth = stdout.columns`（幅頼み）としており、`stdout.columns` が `undefined` の実非TTY環境（`ink-testing-library` の固定 columns=100 では検出できない）でカードが横並びしてしまうバグがあった~~ → **review-changes で発見・修正**。列数1のときは `flexDirection="column"` でレイアウト方向そのものにより縦積みを構造的に保証する形に直した（`src/cli/components/instance-table.tsx`）。回帰テストは Ink の `render()` を直接使い実非TTY相当のstdoutを注入して検証

### FR-03: watch モードを常時 ON にする（優先度: 必須）

- AC-1: `monomi`（引数なし）実行直後から watch（ポーリング）が ON の状態で起動する（起動のために `w` を押す必要がない）
- ~~AC-2: `w` キーで従来通り ON/OFF をトグルできる~~ → **実装中に撤回**。実際に動かして試した結果、手動トグルの必要性が無いと判断し、`w` キー・トグル機能そのものを撤去した（常時 ON のみ。`KeyBindingController` から `w` ハンドリングと `PollingLoop` 依存を削除、footer/ヘルプからも `w watch` の表記を削除）
- AC-3: ヘッダーの `● watching` 表示が起動直後から出る

### FR-04: 詳細ビュー（Agent View Lv.1）表示中のショートカットヒント切替と操作無効化（優先度: 必須）

実際にカード型 UI を触って確認した結果、詳細ビューに入ってもフッターのショートカットヒントが一覧表示用のまま変わらないことが判明し、追加した要件。

- AC-1: 一覧表示中のフッターは `1-5 filter j/k ↑↓ move ↵ detail esc back ? help q quit`（`w watch` は FR-03 により削除済み）
- AC-2: 詳細表示中のフッターは `esc back ? help q quit` のみに切り替わる（フィルタ・移動・詳細を開く操作のヒントは表示しない）
- AC-3: 詳細表示中は `1`-`5`（フィルタ）・`j`/`k`/`↑`/`↓`（カーソル移動）・`Enter`（詳細を開く）の入力を無視する（`esc`/`?`/`q` は引き続き有効）。誤操作で裏の一覧状態が変わるのを防ぐ
- AC-4: `KeyBindingController.handleKey` は現在の `viewMode`（`'list' | 'detail'`）を引数として受け取り、内部 state としては持たない（既存の「薄い写像」規約を維持）

### FR-05: 詳細ビュー（Agent View Lv.1）の自動更新（優先度: 必須）

複数端末で実運用してみた結果、詳細ビューを開いたまま status/recent events の変化に気づけないとの指摘により追加。

- AC-1: 詳細ビュー表示中、一覧と同じ間隔（`pollIntervalMs`）で自動的に再取得され、status・recent events が更新される
- AC-2: 詳細ビューを開いた直後の初回表示（現状維持）に加え、以後は打鍵不要で自動更新される
- AC-3: 詳細ビューを閉じる（`esc` で一覧へ戻る）と自動更新は停止する（無駄なポーリングをしない。コンポーネントのアンマウントで確実に止める）
- AC-4: 取得失敗時は一覧側の既存方針に倣う（直前の表示を保持し、失敗を握りつぶさない範囲でエラー表示は最小限に留める）
- AC-5: ポーリングの実装ロジックを重複させない。既存の一覧側ポーリング機構（`PollingLoop` 相当）を再利用可能な形にする（`listInstances` 固有の実装になっている場合は、任意の非同期取得関数に対応できる形へ一般化する）

## 非機能要件

- 依存追加なし（Ink 標準機能 `flexWrap`/`useStdout` のみで実現する）
- ターミナルのリサイズへの追従は Ink の再レイアウトに委ねる（本リリースでは追加のリサイズハンドリング実装はしない）

## スコープ外（release-4 では実装しない）

- ヘッダー・フィルタバー・フッターへのボーダー付与（カード部分のみの変更に留める）
- 詳細ビュー（Agent View Lv.1）の視覚的なレイアウト変更（FR-05 は自動更新という挙動追加であり、見た目のレイアウトは変えない）
- ファジー検索・ソート循環・デバイスフィルタ循環（引き続き対象外）
