# release-14-pipeline-performance — 要件定義

- リリース識別子: `release-14-pipeline-performance`
- ステータス: 確定
- 作成日: 2026-07-06
- 対応する設計: release-13 の run-release 初回ライブ E2E の実測分析（per-agent タイムライン）。既存課題 `docs/known-issues.md` P5・P6・P7・N2 の解消

## 背景と目的

release-13 で run-release の初回ライブ無人実行（壁打ち確定→PR 作成）が完走したが、所要 1h14m のタイムライン分析により、全体の約 36%（約 27 分）が 3 つの構造的ボトルネック（全差分フル再レビュー・無関係文書への全文整合パス・整形漏れによる Gate1 定型発火）に費やされていることが判明した。また `/workflows` 表示にエージェントが構造上割り当てられない空フェーズが並び、未使用エージェントが展開されたように見える紛らわしさも確認された。

本リリースはこれら実測に基づく 4 課題（P5・P6・P7・N2）を解消し、パイプラインの実行時間と表示の明瞭さを改善する。あわせて、release-12 で「未消費の予約値」とレビュー指摘されていた `config.diffPathScope` を実装済みフィールドへ昇格させる。

## スコープの確定（壁打ちでの決定事項）

| 論点                                 | 決定                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 実施プロセス                         | **本リリースは run-release による無人実行を使わず、従来の段階実行（implement-feature→review-changes→sync-docs→release-check→コミット→PR）で行う**。変更対象がパイプラインエンジン自身（review-changes.js・sync-docs.js）であり、無人実行では「実装フェーズで書き換えた直後の未検証エンジンコードを、同一実行の Gate2・文書同期フェーズがネスト `workflow()` 経由で実行する」自己変更ハザードが生じるため（`workflow()` の name/scriptPath 解決は呼び出し時点のファイルを読む）。release-12 と同じ前例 |
| diffPathScope の意味論               | config の値はプロジェクト既定（モノレポ用の恒常フィルタ）、args の値は呼び出し単位の上書き（args 優先）。両方 null なら従来どおり全差分                                                                                                                                                                                                                                                                                                                                                               |
| 再レビューのスコープ対象             | Gate2 の修正後の**再レビューのみ**絞る（初回レビューは全差分のまま）。スコープは fix 前スナップショットとの差分ファイル一覧。全 4 次元は維持（次元は削らない）                                                                                                                                                                                                                                                                                                                                        |
| sync-docs トリアージの誤スキップ対策 | 判定は包含側に倒す（「明確に無関係」のときだけスキップ）。スキップした対象と理由を戻り値に明示し、事後検証可能にする                                                                                                                                                                                                                                                                                                                                                                                  |

## 機能要件

### FR-01: `review-changes.js` の diffPathScope 実装（優先度: 必須）

- 場所: `.claude/workflows/review-changes.js`、`.claude/config.schema.json`（description 更新のみ）
- 対応する既存課題: `docs/known-issues.md` P5（前段）
- AC-1: `args.diffPathScope`（文字列または文字列配列）が指定された場合、レビュー対象の差分取得（`git diff` 系コマンドと未コミット変更の把握）を当該パスに限定する指示をレビューエージェントのプロンプトに反映する。未指定時は `config.diffPathScope` を既定値として使い、それも null なら従来どおり全差分
- AC-2: `config.schema.json` の `diffPathScope` の description から「現状はどのエンジンも参照しておらず〜予約値」の注記を削除し、実装済みの説明（config=プロジェクト既定、args=呼び出し単位の上書き）へ更新する
- AC-3: スコープが指定された実行では、戻り値にスコープ内容（対象パス一覧）を含め、何が絞られたかを事後確認できるようにする

### FR-02: run-release Gate2 再レビューのスコープ縮小（優先度: 必須）

- 場所: `.claude/workflows/run-release.js`（Gate2 レビューループ）
- 対応する既存課題: `docs/known-issues.md` P5
- AC-1: Gate2 の修正後の再レビュー（`workflow('review-changes', ...)` の 2 回目以降の呼び出し）で、fix 前スナップショットとの差分ファイル一覧（既存のテスト不変ガードが使う snapshot 機構から取得）を `args.diffPathScope` として渡す
- AC-2: 初回レビュー（Gate2 の 1 回目）は従来どおり全差分を対象とする（スコープを渡さない）
- AC-3: 差分ファイル一覧の取得に失敗した場合（スナップショット参照が得られない等）は、安全側に倒してスコープなし（全差分）で再レビューする

### FR-03: implement-feature 実装エージェントの整形自走（優先度: 必須）

- 場所: `.claude/workflows/implement-feature.js`（実装エージェントのプロンプト）
- 対応する既存課題: `docs/known-issues.md` P6
- AC-1: 実装エージェントのプロンプトに「作業完了前に、config.checks のうち format・lint に相当する検査コマンドを実行し、自分が編集したファイルに起因する失敗があれば整形・修正してから完了すること」の指示を追加する（検査コマンドは config 経由で埋め込み、プロジェクト固有値をプロンプト本文にハードコードしない）
- AC-2: プロジェクト中立性を維持する（`grep -rniE 'monomi|/opt/dev|pnpm' .claude/workflows .claude/commands` のヒットが引き続き 0 件）

### FR-04: sync-docs の関連性トリアージによる早期 no-op（優先度: 必須）

- 場所: `.claude/workflows/sync-docs.js`
- 対応する既存課題: `docs/known-issues.md` P7
- AC-1: 差分要約の後、更新エージェント群の起動前に、軽量モデル（`config.models.bootstrap` 相当のティア）による関連性トリアージを 1 回挟む。各同期対象について「差分がこの文書の記述対象に関係するか」を構造化出力（対象パス・relevant・理由）で判定させる
- AC-2: トリアージで「明確に無関係」と判定された対象は更新エージェントを起動せず、戻り値に skip として対象パスと理由を明示する。判定が曖昧な場合は必ず起動側に倒す（誤スキップより誤起動を許容）
- AC-3: トリアージエージェントが null を返した場合は全対象を起動する（従来挙動へのフォールバック。サイレントな全スキップの禁止）

### FR-05: run-release meta.phases の空フェーズ整理（優先度: 必須）

- 場所: `.claude/workflows/run-release.js`（meta.phases）
- 対応する既存課題: `docs/known-issues.md` N2
- AC-1: `meta.phases` から、実作業をネスト `workflow()` に委譲するためエージェントが構造上割り当てられない「実装」「起票」「文書同期」の宣言を削除する（「準備」は config ブートストラップのフォールバック経路で使われるため残置）
- AC-2: `phase()` 呼び出し・エージェントの `phase` 指定と削除後の meta.phases の整合を保つ（削除したタイトルを参照する `phase:` 指定が残らないこと）

### FR-06: エンジン自己変更リリースの運用注記（優先度: 推奨）

- 場所: `docs/development-workflow.md`
- AC-1: run-release 運用節に「`.claude/workflows/*.js` 自身を変更するリリースは、無人実行中に未検証の新エンジンコードをネスト呼び出しが実行する自己変更ハザードがあるため、従来の段階実行で行う」旨の注記を追記する（release-12・release-14 の前例を根拠として記載）

### FR-07: 既知課題の解決済みログ移動（優先度: 必須）

- 場所: `docs/known-issues.md`
- AC-1: P5・P6・P7・N2 を「未解決（バックログ）」から「解決済みログ」へ移動し、解決リリースを `release-14（対応 FR 番号つき）` で記載する

## 非機能要件

- **後方互換**: diffPathScope・トリアージとも未指定/失敗時は従来挙動（全差分・全文書更新）へフォールバックし、既存の呼び出し方を壊さない
- **プロジェクト中立性**: エンジン `.js` 本文へのプロジェクト固有値の追加なし（FR-03 AC-2 の grep 検証）
- **既存品質**: `pnpm run lint` / `format:check` / `test`（625 件） / `build` 全パス

## スコープ外（release-14 では実装しない）

- レビューモデル（Opus 4.6 固定）の変更（review:arch 10.9 分の主因だが、再現性優先のピン留めはユーザーの明示決定。変更する場合は別途意思決定）
- record-known-issues の解決済みログ移動の自動化（現状どおり実装エージェント/提案 diff で行う）
- implement-feature 設計フェーズの高速化（設計 6 分は品質とのトレードオフで現状許容）
- テンプレートリポジトリ抽出（Phase 3）・他プロジェクト導入検証（Phase 4）

## 次のステップ

本リリースはエンジン自己変更のため run-release を使わず、従来の段階実行で行う（スコープ確定表参照）:

```
Workflow({name: "implement-feature", args: {release: "release-14-pipeline-performance"}})
→ review-changes → sync-docs → release-check → /logical-commits → push・PR 作成
```
